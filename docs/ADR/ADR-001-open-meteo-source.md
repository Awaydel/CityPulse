# ADR-001: Выбор Open-Meteo как единственного источника данных

## Статус

✅ Принято

## Контекст

По ТЗ предлагалось использовать два источника:
- **OpenAQ** — реальные измерения со станций мониторинга
- **Open-Meteo** — модельные данные погоды и качества воздуха

## Решение

Используем **только Open-Meteo API** для получения данных о погоде и качестве воздуха.

## Обоснование

| Критерий | OpenAQ | Open-Meteo | Выбор |
|----------|--------|------------|-------|
| Авторизация | Требуется API-ключ | Не требуется | ✅ Open-Meteo |
| Покрытие городов | Зависит от станций | Глобальное (модель) | ✅ Open-Meteo |
| Единый API | Только воздух | Погода + воздух | ✅ Open-Meteo |
| Стабильность | Зависит от станций | Всегда доступны данные | ✅ Open-Meteo |
| Сложность интеграции | Высокая (агрегация по станциям) | Низкая | ✅ Open-Meteo |

## Последствия

### Положительные
- Упрощённая архитектура ETL
- Не нужна регистрация и управление API-ключами
- Гарантированное покрытие для любого города

### Отрицательные
- Данные модельные, а не реальные измерения
- Нет детализации по станциям внутри города
- Ограниченный набор загрязнителей (нет NO₂, O₃)

## Альтернативы

1. **Использовать оба API** — увеличивает сложность, требует сопоставление данных
2. **Только OpenAQ** — не все города покрыты станциями
